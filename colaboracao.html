Visao Geral

Somos 3 agentes trabalhando no mesmo repositorio: dois humanos especialistas e um assistente de IA rodando localmente em cada maquina.

🎨 Dev Frontend
Interface & UX
  • Pagina publica por tenant (LP + agendamento)
  • Painel admin (dashboard, agenda, CRUDs)
  • Design system em src/components/ui
  • Hooks de UI e consumo de API
  • Responsividade mobile-first
⚙️ Dev Backend
API, Dados & Regras
  • Schema do Supabase + migrations
  • RLS (Row Level Security) multi-tenant
  • API Routes em src/app/api
  • Servicos (disponibilidade, comissao, audit log)
  • Zod validators + types gerados
🤖 Claude Code (cada dev)
Assistente na maquina local
  • Le o repo como qualquer dev (via git clone)
  • Gera codigo, refatora, escreve testes
  • Le CLAUDE.md e .claude/rules/ antes de agir
  • Nunca faz push sem aprovacao humana
  • Cada dev tem o seu Claude local — nao compartilha sessao
💡
Principio chave: a fonte unica da verdade e o main no GitHub. Tudo que nao esta no Git nao existe. Claude e ferramentas locais assistem o desenvolvimento, mas o fluxo oficial sempre passa por branch + PR + review + merge.

Divisao Front / Back

Quem toca quais pastas. Ambos podem ler tudo; mudar tem dono.

Frontend Areas de dominio

  • src/app/(public)/[slug]/* — pagina publica do tenant
  • src/app/admin/* — paineis (exceto API)
  • src/components/ui/* — design system
  • src/components/booking/* — fluxo de agendamento
  • src/components/admin/* — UI do painel
  • src/hooks/ui/* — hooks de estado de UI
  • tailwind.config.ts, app/globals.css

Backend Areas de dominio

  • src/app/api/** — todas as API Routes
  • src/lib/supabase/* — clients e helpers
  • src/lib/services/* — regras de negocio
  • src/lib/validations/* — Zod schemas
  • src/lib/api/auth.ts — requireAdmin, etc.
  • supabase/migrations/* — SQL
  • src/types/database.ts — gerado do Supabase

Ambos Zonas compartilhadas

  • src/hooks/useAuth.ts, useTenant.ts — coordenar em PR
  • src/types/* (exceto database.ts) — tipos de dominio
  • CLAUDE.md, .claude/rules/* — regras do projeto
  • README.md, docs
  • Configs raiz: next.config.ts, tsconfig.json, package.json
🤝
Regra de ouro: se sua mudanca mexe numa area do outro, abra PR separado e marque o dono como reviewer. Nunca empurre mudanca de area do outro junto com sua feature.

Setup da Maquina (uma vez)

Execute na ordem. Se algo falhar, pare e pergunte no chat do time antes de forcar.

  1. Instalar Node 20 via nvm 
    # macOS / Linux
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
nvm install 20
nvm use 20
node -v  # deve imprimir v20.x.x
  2. Instalar pnpm 
    npm install -g pnpm
pnpm -v
  3. Instalar GitHub CLI e autenticar 
    # macOS
brew install gh
gh auth login  # escolha GitHub.com → HTTPS → Login with browser
  4. Instalar Docker Desktop

    Baixe em docker.com/products/docker-desktop. Abra pelo menos uma vez para aceitar termos.

  5. Instalar Supabase CLI 
    brew install supabase/tap/supabase
supabase -v
  6. Instalar Vercel CLI (opcional mas recomendado) 
    pnpm add -g vercel
vercel login
  7. Instalar Claude Code

    Siga docs.claude.com/claude-code. Apos instalar:

    claude  # abre a CLI; na primeira execucao, pede login

Clonando o Repositorio

O repo oficial vive no GitHub da organizacao. Nao clone de fork — use o remoto oficial.

  1. Clonar via HTTPS (recomendado com gh auth) 
    cd ~/Projetos
gh repo clone ViniciusMolinari/dunee001
cd dunee001
  2. Instalar dependencias 
    pnpm install
  3. Subir o Supabase local 
    supabase start  # baixa imagens Docker na primeira vez (~5min)

    Ao final, imprime as URLs locais (API, Studio, DB). Anote a anon key e service_role key — vao pro .env.local.

  4. Aplicar migrations no banco local 
    supabase db reset  # aplica todas as migrations do zero
  5. Criar o .env.local

    Copie .env.example → .env.local e preencha (ver secao 7).

  6. Rodar o dev server 
    pnpm dev  # abre em http://localhost:3000

Configurando Claude Code

Cada dev roda Claude Code localmente, na sua maquina, dentro da pasta do projeto. O Claude le o repo via filesystem — nao precisa de servidor, nao precisa de VPS, nao compartilha sessao.

Como funciona a integracao Claude + Git

Claude Code ja roda dentro da pasta do repositorio. Ele:

  • Le arquivos com as mesmas permissoes que voce;
  • Executa comandos git (git status, git diff, gh pr create);
  • Respeita o CLAUDE.md e .claude/rules/ como contexto obrigatorio;
  • Nunca faz push sem que voce mande explicitamente;
  • Nao tem acesso a credenciais que nao estao no seu .env.local.

Primeira execucao

  1. Entrar na pasta do projeto 
    cd ~/Projetos/dunee001
  2. Abrir o Claude Code 
    claude

    Na primeira vez, ele pergunta sobre permissoes (leitura, edicao, bash). Aceite as padroes.

  3. Deixar o Claude ler o contexto do projeto

    Peca: "Leia o CLAUDE.md e o .claude/rules/ antes de comecar qualquer tarefa." Ele carrega as regras do projeto na sessao.

  4. Fixar o modelo e estilo

    Use /model para escolher o modelo (recomendado: o mais capaz disponivel). Use /config para ajustar preferencias locais.

🚨
Regras ao trabalhar com Claude:
1. Sempre peca o plano antes de implementar ("Apresente o plano e aguarde aprovacao").
2. Nunca deixe o Claude dar git push direto — revise o diff primeiro.
3. Nunca deixe o Claude mexer no .env.local — secrets sao pessoais.
4. Se o Claude sugerir workaround esquisito, peca a solucao profissional.

Permissoes sugeridas (.claude/settings.local.json)

Cada dev cria o seu. Nao versionar este arquivo.

{
  "permissions": {
    "allow": [
      "Bash(pnpm *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(git log *)",
      "Bash(supabase *)",
      "Bash(gh pr *)"
    ],
    "ask": [
      "Bash(git push *)",
      "Bash(git reset *)",
      "Bash(rm *)"
    ]
  }
}

Fluxo Git

Trunk-based simplificado. main sempre verde, sempre deployavel. Features em branches curtas (1-3 dias max).

Arvore de branches

main # producao, protegida ├── feat/agenda-ui # dev front ├── feat/appointments-api # dev back ├── feat/multi-tenant-middleware # dev back ├── feat/booking-flow-public # dev front ├── fix/availability-timezone # bug fix └── chore/update-deps # manutencao

Nomenclatura de branches

  • feat/* — nova feature
  • fix/* — correcao de bug
  • chore/* — manutencao, deps, config
  • refactor/* — refactor sem mudar comportamento
  • docs/* — apenas documentacao
  • test/* — apenas testes

Use kebab-case curto e descritivo.

Conventional Commits

Padrao obrigatorio para mensagens de commit.

# Formato
<tipo>(<escopo>): <descricao curta>

# Exemplos
feat(agenda): add week view
fix(booking): correct timezone in slots
chore(deps): bump next to 16.1
refactor(api): extract availability service
docs(readme): update local setup steps

Ciclo de trabalho (dia-a-dia)

  1. Atualize main e crie sua branch 
    git checkout main
git pull origin main
git checkout -b feat/nome-da-feature
  2. Desenvolva

    Use Claude Code para gerar/refatorar. Commits pequenos e frequentes (cada commit = uma mudanca coerente).

  3. Commit (pre-commit hook roda validate-rules.sh) 
    git add src/app/admin/agenda/page.tsx
git commit -m "feat(agenda): add filter by professional"

    Se o hook bloquear, corrija a causa. Nunca use --no-verify.

  4. Push da branch 
    git push -u origin feat/nome-da-feature
  5. Abrir PR 
    gh pr create --fill  # usa commits como descricao

    Vercel cria preview automaticamente. CI roda typecheck + lint + testes.

⚠️
Nunca force push em main. Em sua propria branch, --force-with-lease e OK apos rebase. Em branches de outra pessoa, nunca.

Pull Requests & Reviews

Todo codigo entra em main via PR. Nada direto. Review de pelo menos 1 pessoa.

Template de PR (auto via .github/pull_request_template.md)

## Resumo
- Bullet 1
- Bullet 2

## Mudancas principais
- Arquivo X: o que mudou
- Arquivo Y: o que mudou

## Como testar
1. Passo 1
2. Passo 2

## Checklist
- [ ] Dentro dos limites de linha (page 300, componente 200, hook/service 250, API 150)
- [ ] Logica de negocio separada da UI
- [ ] Sem duplicacao de codigo existente
- [ ] Zod validators para inputs de API
- [ ] Logger em todos os catch server-side
- [ ] Preview Vercel testado

Regras de merge

  • CI verde (typecheck + lint + test)
  • 1 aprovacao do owner da area ou do outro dev
  • Sem conflitos com main
  • Preview Vercel funcionando
  • Merge via Squash & Merge (historico limpo em main)

Tamanho ideal de PR

  • Ate 400 linhas de diff — revisa em 15 min
  • 400-800 linhas — aceitavel, revisa em 30 min
  • >800 linhas — quebre em PRs menores

PR gigante = review ruim = bug em producao.

Contrato Front x Back

Como front e back combinam interface sem virar telefone-sem-fio.

1. Types gerados do Supabase = fonte unica

Backend roda supabase gen types typescript --local > src/types/database.ts sempre que muda schema. Esse arquivo e commitado. Frontend importa daqui — nunca redefine tipo.

// Correto (frontend)
import type { Database } from '@/types/database'
type Appointment = Database['public']['Tables']['appointments']['Row']

// Errado — nao redefina tipo de tabela
interface Appointment { id: string; ... }

2. Validators Zod = contrato de API

Backend define schema em src/lib/validations/<recurso>.ts. Front importa o mesmo schema pra validar o form antes de mandar. Um contrato, zero divergencia.

// src/lib/validations/appointments.ts (BACK define)
export const createAppointmentSchema = z.object({
  serviceId: z.string().uuid(),
  professionalId: z.string().uuid(),
  startAt: z.string().datetime(),
  clientName: z.string().min(2),
  clientPhone: z.string().min(10),
})

// Front e back usam o mesmo import
import { createAppointmentSchema } from '@/lib/validations/appointments'

3. Mock de API durante desenvolvimento paralelo

Se o back ainda nao terminou a API POST /api/appointments, o front nao espera. Usa MSW (Mock Service Worker) em src/mocks/ para mockar a resposta combinada via Zod schema. Quando back entrega, desliga o mock.

4. Canal de combinado

Toda nova rota de API entra numa issue "API Contract: POST /api/xxx" com:

  • Payload (link para Zod schema)
  • Response shape
  • Status codes esperados
  • Regras de autorizacao

Front + back concordam no issue antes de codar.

Regras de Qualidade

Automatizadas no pre-commit e na CI. Leia o CLAUDE.md do projeto — essas regras sao obrigatorias.

Limites de linhas

  • Page: 300
  • Componente: 200
  • Hook / Service: 250
  • API Route: 150

Estourou? Extraia hook, sub-componente ou service.

Anti-patterns bloqueantes

  • <img> → next/image
  • console.error server-side → logger
  • Math.random() para ID → generateId()
  • .toFixed().replace() moeda → formatMoney()
  • isAdmin() local → requireAdmin()
  • Cache manual de data → date-fns
  • Fallback de secret → throw
  • catch vazio → logger
✅
Tudo isso e validado automaticamente no commit. O pre-commit roda validate-rules.sh. Se falhar, leia a mensagem e corrija — nunca pule.

# Push + PR

# Ver logs do Supabase

supabase logs

# Abrir Supabase Studio
open http://127.0.0.1:54323

# Ver diff staged
git diff --cached

# Ver PR checks
gh pr checks
gh pr view --web

# Typecheck
pnpm typecheck

# Lint + fix
pnpm lint --fix

# Testes
pnpm test
pnpm test:e2e

Claude Code — prompts uteis

  • Iniciar tarefa: "Leia CLAUDE.md e .claude/rules/, explore a pasta X, apresente plano e aguarde aprovacao."
  • Refatorar: "Refatore este componente mantendo comportamento identico. Respeite limite de 200 linhas."
  • Criar feature: "Implemente a API /api/xxx conforme o Zod schema em src/lib/validations/xxx.ts."
  • Debug: "Investigue o erro X, liste as causas possiveis com evidencias do codigo."
  • Code review: "Revise o diff da branch atual contra os checklists do CLAUDE.md."

Troubleshooting

Problemas comuns e solucao rapida.

🔄 supabase start trava ou falha

supabase stop
docker ps                  # confirme que Docker esta rodando
supabase start --debug

🔄 Erro de types apos mudanca de schema

pnpm db:types              # regenera src/types/database.ts
pnpm typecheck

🔄 Pre-commit bloqueando commit

Leia a mensagem — o validate-rules.sh aponta o arquivo e a regra violada. Corrija. Nunca use --no-verify.

🔄 PR com conflito em main

git checkout main
git pull
git checkout feat/minha-branch
git rebase main
# resolva conflitos arquivo por arquivo
git add .
git rebase --continue
git push --force-with-lease

🔄 Claude sugeriu algo que quebra regra do projeto

Responda: "Isso viola a regra X do CLAUDE.md. Proponha alternativa que respeite o limite/padrao." Claude e assistente, nao oraculo — voce manda.
